## 1. AST 语义质量预测器（Surrogate Model）⏳

### 背景
当前 GP 的随机生成、变异、交叉完全依赖启发式规则和权重偏置。引入一个轻量模型后，可以在进入真实 fitness 评估前快速筛选/过滤候选表达式，把算力集中到更合理的方向。

### 目标
- 预测表达式的潜在质量（0~1 或 fitness 回归）。
- 过滤明显无效或过度复杂的表达式。
- 不过度拟合历史数据，不替代 GP 的探索能力。

### 方案概述
1. **特征工程**：从 AST 提取固定长度语义向量。
   - 结构特征：depth、nodes、calls、binops、compares、ifexps、boolops。
   - 原语频率：ts_xxx、math_xxx、var_xxx 出现次数。
   - 语义比例：ts_ratio、math_ratio、logic_ratio、var_diversity。
2. **模型**：LightGBM 回归器（小模型、可解释、无需 GPU）。
3. **数据**：GP 运行过程中累积的 `(expr_str, fitness)` 对，外加 SQLite 缓存历史数据。
4. **使用方式**：作为可选 `surrogate_model` 注入 `GPEngine`。

### 接入点
- `_random_tree()` / `_random_tree_explore()`：生成后预测，低质量则重试。
- `_mutate()`：变异后预测，过低则保留原个体。
- `_crossover()`：交叉后预测，过低则回退父代。

### 防过拟合策略
- 小模型：`num_leaves=15`、`max_depth=5`。
- 早停：`early_stopping(20)`。
- 样本过滤：剔除 `fitness == -999` 的无效样本。
- 特征/样本子采样：`feature_fraction=0.8`、`bagging_fraction=0.8`。
- 标签分位数化：降低极端 fitness 的影响。
- 冷启动保护：样本 < 100 时不启用模型。
- 定期重训练：每 `N` 代用累积数据重新拟合。

### 验证方式
- 对比开启/关闭 surrogate 后的 `valid_count` 和 `best_fitness` 收敛曲线。
- 观察模型特征重要性，确认与已知有效因子原语一致。

## 2. 权重归一化保护 ✅

当前 `_update_direction_weights()` 使用 EMA 更新 `var_weights` / `ts_weights` / `math_weights`，但缺少归一化。长期运行后权重可能发散或归零，影响探索多样性。

建议：
- 每次 EMA 更新后，对每组权重做 softmax 或除以总和归一化。
- 保留一个最小权重值（如 0.01），确保任何方向都不会被完全关闭。

**[完成] 2026-07-05** 实现：新增 `GPEngine._normalize_weights(weights, min_w=0.05)` 静态方法，在 `_update_direction_weights()` 中 EMA 更新后调用，对每组权重做归一化并保留 `min_w=0.05` 下限。

---

## 3. 停滞参数恢复 ✅

当前 `_adapt_operators()` 在停滞时会提升 `mutation_prob`、降低 `crossover_prob`，并翻倍 `random_inject_count`。但改善后只恢复了前两者，`random_inject_count` 没有恢复。

建议：
- 在检测到改善时，将 `random_inject_count` 恢复为初始值。

**[完成] 2026-07-05** 实现：`GPEngine.__init__` 中保存 `self._save_random_inject = self.random_inject_count`；`_adapt_operators()` 检测到改善时调用 `self.random_inject_count = self._save_random_inject` 恢复。同时 `_save_crossover_prob` / `_save_mutation_prob` 也采用同样模式。

---

## 4. SQLite 缓存指纹设计说明 ℹ️

当前 `fitness_hash` 基于数据形状和 `parsimony_penalty`。

由于 `cache_db` 是按独立项目/数据集维护的（回测品种、时间范围固定），同一项目内 shape 相同即意味着数据集相同，因此当前 fingerprint 已足够。无需加入数据内容 hash。

---

## 5. callback 接口 ⏳

当前 `run(verbose=True)` 使用 logger 输出进度。外部模块（如 LLM 智能体）难以实时获取状态。

建议：
- 增加 `callback: Callable[[int, Individual, Dict], None]` 参数。
- 每代调用一次，传入 `(generation, best_individual, stats)`。

## 6. 语义去重 / AST 规范化 ✅

已在 `engine.py` 中实现 `_canonicalize_ast()`。

实现内容：
- 交换律排序：`a + b` 与 `b + a` 规范化后相同。
- 纯常数折叠：`1 + 2` → `3`。
- 安全边界：不处理非交换函数参数（如 `ts_corr(a,b)` 与 `ts_corr(b,a)` 视为不同）。
- 接入点：`_evaluate_individual()` 和 `_evaluate_parallel()` 用规范化 AST 生成缓存 key；`ind.expression_str` 仍保留原始可读表达式用于报告展示。

## 7. 多实例多线程安全重构 ✅ (2026-07-05 额外完成)

不在原计划内，但在使用过程中识别出多线程隐患，已一并重构：

**重构前问题**：
- `engine.py` 模块级全局变量 `_TREE_GEN_CONFIG`：多实例共享会互相污染
- `random` 模块级状态：多线程下不可复现
- `_CANONICALIZE_MEMO` (id-based)：AST 节点 id 跨 garbage collection 会重用，可能错命中
- `_update_direction_weights` 缺少归一化
- `_adapt_operators` 停滞恢复时漏掉 `random_inject_count`
- `_lexicase_select` 负数 fitness 阈值计算错误
- `_expr_signature` 变量未按频率排序

**重构后**：
- `TreeGenConfig` 新增 `rng: random.Random` 字段，每个 `GPEngine` 实例持有独立 rng
- 所有 `_random_*` 函数接收 `cfg: TreeGenConfig` 参数，使用 `cfg.rng`
- `_CANONICALIZE_MEMO` 改为 `_CANONICALIZE_STR_MEMO`，以 expr_str 做 key
- 新增 `_canonicalize_key(tree, expr_str)` 直接返回字符串
- `_random_tree_explore(user_cfg, max_depth)` 不再切换全局，构建临时 explore_cfg 共享 user_cfg.rng
- `GPEngine.__init__` 创建实例 rng，调用 `_build_tree_config(tree_gen_config, rng)` 注入
- 新增 `_normalize_weights` 静态方法，EMA 更新后归一化
- `_adapt_operators` 改善时恢复 `random_inject_count`
- `_lexicase_select` 阈值改为 `best_v - abs(best_v) * 0.2` (修复负数场景)
- `_expr_signature` 用 `Counter` 按出现频率排序变量

**验证**：
- `tmp/test_gp_v5_e2e.py` 端到端测试通过：单实例 run、多实例多线程并行、相同种子可复现。

## 8. 代码审查 P0/P1 修复 ✅ (2026-07-06)

对 2026-07-05 重构再做深度审查时发现 8 个问题（1 个 P0 + 7 个 P1），全部已修复：

### P0-1: 交叉深度保护完全失效 ✅
- **问题**：`_crossover` 中 `test_tree = copy.deepcopy(child_tree)` 后调用 `_replace_subtree(test_tree, n1, ...)`，但 `n1` 来自原 `child_tree`，不在 `test_tree` 中，`_replace_subtree` 永远返回 False，深度保护完全失效。交叉产物可任意超过 `max_depth`，多代后树深度爆炸。
- **修复**：直接在 `child_tree` 上替换，超深则回滚到 `backup`；每次循环重新收集 `candidates1_all`（回滚后节点对象已变）。

### P1-1: `_CANONICALIZE_STR_MEMO` 仍模块级共享 ✅
- **问题**：重构说明声称消除模块级全局变量，但 `_CANONICALIZE_STR_MEMO` 仍是模块级 dict，多线程并发写有 check-then-set 竞态，且无大小上限会内存泄漏。
- **修复**：memo + `threading.Lock` 移到 `GPEngine` 实例（`self._canonicalize_memo` / `self._canonicalize_lock`），`_canonicalize_key` 接收 memo+lock 参数。

### P1-2: 方向快照按全局偏移，新签名记录永不被处理 ✅
- **问题**：`new_fits = fits[self._direction_snapshot:]` 用全局偏移套到每个签名上，新签名记录数小于 snapshot 时永远取不到，破坏方向权重闭环。
- **修复**：新增 `self._direction_per_sig_snapshot: Dict[str, int]`，按签名独立记录偏移。

### P1-3: Lexicase 的 `sig` 未使用，等价于普通锦标赛 ✅
- **问题**：`for sig in cases: ...` 循环体里 `sig` 从未被引用，只是反复用整体 fitness 做 20% 容差过滤，没有方向多样性选择压力。
- **修复**：改为从每个签名的 `best_expr` 反查对应个体，组成"方向代表池"，加权选择。

### P1-4: nan/inf fitness 未检查，污染选择和缓存 ✅
- **问题**：`if fitness < -998.0` 在 nan/inf 时为 False，会通过。nan 让 `max()` 行为未定义，inf 让错误个体永远胜出。
- **修复**：改用 `if not np.isfinite(fitness) or fitness < -998.0`。

### P1-5: `np.random.seed(random_seed)` 设全局状态 ✅
- **问题**：多实例多线程下，后创建的实例会覆盖前一个的 numpy 随机状态，前一个实例的 `fitness_calc` 行为不可复现。
- **修复**：完全移除 `np.random.seed(random_seed)` 调用，由 `fitness_calc` 自行管理随机数（调用方应传入 `np.random.Generator`）。

### P1-6: `max_depth<2` 时 `randint(2, max_depth)` 崩溃 ✅
- **问题**：`rng.randint(2, max_depth)` 在 `max_depth=1` 或 `0` 时抛 `ValueError: empty range for randrange()`。
- **修复**：改为 `rng.randint(2, max(2, max_depth))`，两处（`_random_tree` 和 `_random_tree_explore`）。

### P1-7: 变异不校验产物深度，与 P0-1 叠加导致深度无界增长 ✅
- **问题**：`_mutate` 在变异后直接返回，不校验产物深度。子树替换后总深度可能超过 `max_depth`。
- **修复**：`_mutate` 统一在变异后加 `if ast_depth(new_tree) > self.max_depth: new_tree = copy.deepcopy(individual.tree)`，越深则放弃变异。

### 验证

`tmp/test_gp_v5_regression.py` 8 项回归测试全部通过：
1. 单实例完整 GP run
2. 多实例多线程并行
3. 可复现性（相同种子→相同 best fitness 10482.773182，表达式一致）
4. **P0-1 交叉深度保护**：种群 50 个个体全部深度 ≤ 5（关键验证）
5. **P1-6 max_depth=1 不崩溃**
6. **P1-4 nan/inf fitness 被拦截为 -999**
7. **P1-1 实例级 memo 隔离**
8. Lexicase 选择不崩溃
