## 1. 代码修改与注释规范

### 1.1 修改代码时必须补充注释

**基本原则：**
- 任何逻辑修改都必须添加注释说明
- 注释要解释"为什么修改"，而不是"改了什么"
- 保留历史修改记录，便于追溯

### 1.2 注释格式规范

参考项目现有风格，使用以下格式：

```python
# [修改类型] YYYY-MM-DD HH-MM 简短描述
# 详细说明修改原因、逻辑依据
# （可选）新旧实现对比
```

**修改类型标记：**
- `[修复]` - Bug 修复
- `[优化]` - 性能/代码质量优化
- `[新增]` - 新增功能
- `[重构]` - 代码重构
- `[调整]` - 参数/逻辑调整

**示例：**
```python
# [修复] 2026-05-13 14-30 thr_mean/thr_med 不再在此处用全序列均值实现，改为 expanding 方式(无前瞻偏差)
# 旧实现 'thr_mean': lambda x: np.where(x > np.nanmean(x), 1.0, 0.0) 用了全序列均值含未来数据
# 新实现移到 evaluate() 中，调用 _expanding_mean / _expanding_median
THRESHOLD_FUNCS = {
    'thr_0': lambda x: np.where(x > 0, 1.0, 0.0),
    'thr_mean': None,
    'thr_med': None,
}
```

### 1.3 函数/类文档字符串

- 新增函数/类必须添加 docstring
- 说明用途、参数、返回值
- 复杂逻辑添加 inline 注释

## 2. 工作流程

### 2.1 讨论阶段先不修改代码
- 收到需求后，先分析并总结方案
- 与用户确认技术路径后再实施
- 避免直接修改导致返工

### 2.2 修改前先理解现有代码
- 阅读相关模块的 AI.md 文档
- 查看现有代码的实现逻辑
- 遵循项目的编码风格和架构设计

---

## 3. 其他注意事项

### 3.1 不删除现有注释
- 除非函数逻辑彻底改变，否则保留原有注释
- 历史注释有助于理解代码演进过程

### 3.2 变更透明
- 所有修改都要有据可查
- 注释要说明修改的上下文和决策依据

### 3.3 安全第一
- 不提交敏感信息
- 遵循安全最佳实践

## 4. AI 助手工作流

```
1. 阅读 AI_INDEX.md 了解项目结构
2. 阅读本 AGENTS.md 遵循开发规范
3. 分析任务需求，总结方案
4. 与用户确认方案后再修改代码
5. 修改时补充规范的注释
6. 验证修改正确性
```
